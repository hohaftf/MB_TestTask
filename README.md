# MB_TestTask

Репозиторий содержит 2 папки: отдельная папка для каждого тестового задания. 
Ниже комментарии к решениям.

## Библиотека вычисления площади

Сделано 2 проекта.
MindboxAreaCalculator учитывает легкость добавления новых фигур.
Вся логика фигуры реализована в одном классе: и вычисление площади, и валидация.
Однако это противоречит принципу разделения ответственности, поэтому также был сделан проект MindboxAreaCalculator2, в котором в отдельные классы вынесена валидация.

В первом проекте валидации параметров и фигуры происходит в конструкторе. Я так сделал для того, чтобы в случае расширения функциональности не пришлось дублировать валидацию фигур в каждом методе. Отдавать ее на откуп клиентскому коду тоже нехорошо, ее могут просто не выполнять
Из тех же соображений, но с учетом вынесенной валидации, во втором проекте я сделал создание фигур через фабрику, котора создает невалидированный экземпляр, а затем его проверяет.

В задании написано, что оценится вычисление площади фигуры без знания типа фигуры.
Пример того, как это может сделано, реализован во втором проекте в методе CreateFigure фабрики.
Но надо отметить, что это возможно, только в рамках описанной задачи. В случае произвольных фигур мы сразу получим противоречие. Один параметр может описывать как круг, так и квадрат. Можно привести и другие примеры.

## SQL скрипт

Для таблицы связей по правилам надо было бы создать 2 внешних ключа: на таблицу статей и таблицу  тегов.
Но моё личное мнение, что при правильном программировании они не нужны и только замедляют выполнение запросов.
Этот вопрос относится к правилам внутри конкретной команды разработчиков.
